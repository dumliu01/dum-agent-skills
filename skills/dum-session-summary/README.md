# dum-session-summary

> 把**一次会话的改动**沉淀成标准结构的 Markdown，存进 `docs/modify_history/`——**续接** + **校对设计文档**两用。

## 它做什么

git 知道**改了什么**；但只有会话记得**为什么这么改、试过什么被否决、哪些设计文档因此过时**——后者正是 git 永远拿不到的，也是本技能最大的价值。

**双用途**：
1. **续接**：切换/新开会话时，照这篇就能快速接上——做到哪、怎么继续、从哪个文件入手
2. **校对设计文档**：定期回看时，能据此找出哪些 `docs/方案设计/`、`docs/guides/` 已被改动带偏，需要回头修正

**核心原则**：以 git 为客观事实骨架（取 commit + diff），用会话上下文补"为什么"。

## 何时触发

- ✅ "总结这次会话/把改动记录下来/生成交接文档/方便下次继续"
- ✅ 一段较大的工作刚收尾、准备切换上下文
- ❌ 只想要一条 commit message → 用 git
- ❌ 只想知道改了哪些文件 → `git status`
- ❌ 写面向用户的 release notes（产品视角）

## 它交付什么

| | |
|---|---|
| 路径 | `docs/modify_history/YYYY-MM-DD-<标题>.md` |
| 结构 | 固定 **7 节**：一句话概述 / 背景目标 / 改动清单 / 关键决策与理由 / 当前状态与验证 / 如何续接 / 受影响的设计文档（待校对） |
| 关键节 | "关键决策与理由" 必含**被否决方案**；"受影响的设计文档" 主动 grep 推导，无影响也显式写"无" |

## 跟其它技能怎么衔接

- **上游**接代码改动；**下游**接 [`dum-doc-reconcile`](../dum-doc-reconcile/)——后者据这些记录定期对账 `docs/architecture/` 与 `docs/方案设计/`
- 落点目录由 [`dum-knowledge-base-build`](../dum-knowledge-base-build/) 搭好；它的 `update_docs_index.py` 会把本文档自动收录进 `docs/docs-index.md`
