# Dum Session Summary

## Overview

把**一次会话的改动**沉淀成一篇标准结构的 Markdown，存到 `docs/modify_history/`。两个用途决定了它和普通 commit message / changelog 的区别：

1. **续接（用途①）** —— 切换/新开会话时，照这篇就能快速接上：知道做到哪、怎么继续、从哪个文件入手。
2. **校对设计文档（用途②）** —— 定期回看时，能据此找出哪些 `docs/方案设计`、`docs/guides` 已被改动带偏，需要回头修正，让设计文档持续正确。

**核心原则：以 git 为客观事实骨架，用会话上下文补"为什么"。** git 知道**改了什么**；只有会话记得**为什么这么改、试过什么被否决、哪些设计文档因此过时**——后者正是这篇文档最大的价值，也是 git 永远拿不到的。

## 触发判断

**该触发**：用户说"总结这次会话/把改动记录下来/生成交接文档/记录修改历史/方便下次继续"等；或一段较大的工作刚收尾、准备切换上下文时。

**不该触发**：只是要一条 commit message（用 git）；只想知道"改了哪些文件"（`git status` 即可）；要写面向用户的 release notes（那是产品视角，不是开发续接视角）。

## 工作流（按顺序，跳步要说原因）

### 1. 取数：git 事实 + 会话上下文

- **git 客观面**：取本次会话的 commit（`git log` 自会话起点；起点不确定就问用户，或用"上一篇 modify_history 之后"的提交）+ 未提交改动（`git status` / `git diff --stat`）。这是"改动清单"的事实来源。
- **会话主观面**：从对话里捞出 git 看不到的东西——**关键决策、被否决的备选方案、踩过的坑、未完成项、验证方式**。捞不全就回看本会话再补，不要编。

### 2. 推导"受影响的设计文档"（最关键、最容易漏）

这步是用途②的命脉，必须主动做，不能等用户喂：

1. 列出本次改动触及的**模块/功能/文件路径**关键词。
2. 用这些关键词检索 `docs/方案设计/`、`docs/guides/`、`docs/product/`（grep 模块名、功能名、路径），找出**描述了这块、但描述已和现状不符**的文档。
3. 每条写：`文档路径 — 哪里被带偏了 — 建议怎么改`。
4. **若确认无影响，也要显式写"无"**，不能省略本节（否则回看时分不清是"没影响"还是"忘了查"）。

### 3. 写文档：套用下方 7 节模板

实打实填，**不留 `[TODO]`/`待补充`** 占位。写不出某节就回第 1/2 步把信息找齐。文档讲清楚逻辑即可，**不贴大段示例代码**（接口签名、配置项名、commit hash 可以）。

### 4. 落盘

按下方"命名与位置约定"存盘；`docs/modify_history/` 不存在就创建。索引无需手动维护——`docs/README.md` 的知识库脚本会自动收录（见 [scripts/docs/update_docs_kb.py](../../dum-knowledge-base-build/SKILL.md) 的扫描机制）。

## 文档结构模板（7 节，固定顺序）

```markdown
# YYYY-MM-DD 修改摘要：<一句话标题>

## 一句话概述
这次会话到底做成了什么（1–2 句，给"未来的我"看）。

## 背景 / 目标
为什么做、要解决什么问题、触发点是什么。

## 改动清单
按模块/主题分组列改动文件（新增/修改/删除 + 每项一句话作用）。
末尾附本次关键 commit：`<hash> <message>`。

## 关键决策与理由
做了哪些设计选择、为什么；**被否决的方案也要写**（含否决原因）——这是防止下次重走弯路的关键。

## 当前状态 / 验证
已完成到什么程度；怎么验证的（命令/测试/手测结果，给证据不空说）；遗留 TODO 与已知问题。

## 如何续接（下个会话从这接）  ← 服务用途①
从哪里继续、首选下一步、相关入口文件/命令、需要注意的坑。

## 受影响的设计文档（待校对）  ← 服务用途②
本次改动可能让哪些 docs/方案设计·guides·product 过时：
- `文档路径` — 哪里被带偏 — 建议怎么改
（确认无影响则写"无"。）
```

## 命名与位置约定

- 路径：`docs/modify_history/YYYY-MM-DD-<标题>.md`
- 日期：`YYYY-MM-DD`（用户当前日期，非凭空）。
- 标题：简短描述性短语；空格换 `-`，去掉 `/ \ : * ? " < > |` 等非法字符；可中文。
- 同日多篇：追加 `-2`、`-3`。

## 关键逻辑 / 难点

- **会话边界怎么定**：优先"自本会话开始的全部改动"。无法判定起点时，问用户，或以"上一篇 modify_history 时间点之后"为界——别把历史改动重复计入。
- **取数顺序是 git 先、会话后**：先用 git 锚定确切的文件与 commit（防止漏记/记错），再叠加会话里的理由与取舍。反过来容易凭印象写错文件。
- **用途②靠"受影响设计文档"节兑现**：没有这节，这篇就退化成普通 changelog，定期回看时无从修正设计文档。

## 常见错误

| 错误 | 后果 | 修正 |
|---|---|---|
| 只列"改了什么"，不写"为什么/否决了什么" | 退化成 git log，下次会重走弯路 | 必填"关键决策与理由"，含被否决方案 |
| 漏掉/省略"受影响的设计文档"节 | 用途②失效，设计文档持续漂移 | 主动 grep docs/ 推导；无影响也写"无" |
| 无"如何续接"或写得太泛 | 用途①失效，新会话接不上 | 写到"改哪个文件、跑什么命令"的颗粒度 |
| 文件名/日期格式各篇不一 | 难以排序与定期回看 | 严格 `YYYY-MM-DD-<标题>.md` |
| 凭印象写文件清单 | 记错/漏记改动 | 以 `git diff --stat` 为准 |
| 留 `[TODO]` 占位 | 半成品文档 | 信息找齐再写，不留草稿语 |

## 出口判断

- ✅ `docs/modify_history/YYYY-MM-DD-<标题>.md` 已生成，7 节齐全、无占位草稿。
- ✅ "改动清单"与 `git diff --stat` 一致；附了本次 commit。
- ✅ "关键决策"含被否决方案；"受影响的设计文档"已主动推导（无影响则显式写"无"）。
- ✅ "如何续接"具体到文件/命令颗粒度。
