# 架构 vs 规范：内容归属 taxonomy

拆分一份混合 `ARCHITECTURE.md`、或决定一段内容写进哪份文档时，用本表。

## 一句话准则

> **能从代码读出来的「现状」→ 架构文档；要求开发者遵守的「约定」→ 规范文档。**

- 架构 = 描述性（descriptive）：系统**是**怎么搭的。读者目标是「看懂这个系统」。
- 规范 = 规范性（prescriptive）：动手前**必须知道**的规矩。读者目标是「不违反项目硬规矩地加代码」。

判不准时再问一句：**「这段话删了，新人会看不懂系统，还是会写出不合规的代码？」** 前者→架构，后者→规范。

## 归属总表

| 内容 | 去哪 | 说明 |
|---|---|---|
| 项目使命 / 技术栈 / 进程·分层模型 | 架构 | 系统全貌 |
| 目录结构（含 AUTO-GENERATED 源码树） | 架构 | 代码在哪 |
| 核心模块职责 / 模块间关系 / 依赖图 | 架构 | 怎么组织 |
| 启动流程 / 数据流 / 消息时序 | 架构 | 怎么流转 |
| 对外接口**清单**（当前有哪些 endpoint/WS 消息/IPC 通道） | 架构 | 现状盘点 |
| 数据模型 / 表结构 / schema | 架构 | 持久化现状 |
| 各子系统设计（认证机制、加密体系、计费链路、容器隔离…） | 架构 | 怎么实现的 |
| **命名规范**（文件/包/变量/类型前缀） | 规范 | 写代码的约定 |
| **注释 / docstring 语言**（中文还是英文） | 规范 | 必须遵守 |
| **API 风格约定**（RPC vs REST、URL 格式、统一响应、动词命名） | 规范 | 新接口要遵循 |
| **错误码体系**（分段表 + 新增规则） | 规范 | 加错误码要按段 |
| **日志规范**（级别、结构化字段、脱敏、滚动策略要求） | 规范 | 打日志要遵守 |
| **性能规则**（分包策略、懒加载边界、"禁止静态导入 X"） | 规范 | 硬规则 |
| **错误处理规范**（error wrapping、异常分类、不吞异常） | 规范 | 写法约定 |
| **并发 / 异步规范**（goroutine 生命周期、不阻塞 event loop） | 规范 | 写法约定 |
| **类型 / 注解规范**（禁 any、type hints 必填） | 规范 | 写法约定 |
| **配置规范**（新增配置项放哪、env 命名、默认值规则） | 规范 | 加配置要遵守 |
| **测试规范**（测试放哪、命名、覆盖要求） | 规范 | 写测试要遵守 |
| **提交规范**（commit message 语言/格式） | 规范 | 提交要遵守 |
| **开发速查**（"新增 X → 改哪些文件"表） | 规范 | 动手指引 |

## 灰区与拆法

几个最容易纠结的：

- **接口清单 vs API 风格**：当前**有哪些**接口、各自参数/返回 → 架构（现状盘点）；**新接口该长什么样**（POST-only、URL 格式、code 字段约定）→ 规范。同一个 API 主题在两份文档各占一节，互不重复。
- **日志系统 vs 日志规范**：日志**架构**（双日志、文件结构、按日滚动怎么实现的）→ 架构；打日志**要遵守什么**（哪些级别、必带哪些字段、敏感信息脱敏）→ 规范。实现细节归架构，要求归规范。
- **错误码体系**：通常整段进**规范**（它本质是"加错误码要按这些段、用这个枚举"的约定）。架构里只在相关子系统提一句"错误通过 code 字段返回"。
- **性能架构规范**：名字里有"架构"，但凡是带"**规则：**禁止/必须"的（如"弹窗必须懒加载""禁止直接 import 插件模块"）→ 规范；纯描述当前分包结果的表 → 可留架构的"构建与部署"节。
- **关键约定 / invariants**：是"不能违反的硬规矩" → 规范（或架构里只列、规范里展开）。

## 拆已有混合文档的步骤

1. 列出混合文档的所有一级/二级标题。
2. 逐节按上表打标：A（架构）/ S（规范）/ 拆（一节里两类都有，要切开）。
3. 标 A 的搬进 `<lang>-architecture` 对应章节；标 S 的搬进 `<lang>-specification` 对应章节。
4. 标「拆」的：把描述句留架构、把"必须/禁止/应该"句搬规范，同一主题两边各成一节并互链。
5. 两份都补头部互链、删掉原混合文档（或留一行重定向到两份新文档）。
