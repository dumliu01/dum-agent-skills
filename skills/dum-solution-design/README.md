# dum-solution-design

> 出结构化技术方案：**5 模块**（架构 / 时序 / 关键逻辑 / 接口 / 遗留）+ **文档与代码分离** + 命名/位置约定。

## 它做什么

方案文档容易出三类病：（1）代码塞进设计里，重构后文档变垃圾；（2）少了"被否决方案"和"遗留问题"，下次重走弯路；（3）没有时序图，调用顺序看不出来。本技能把方案文档强制成**5 模块固定结构**，并**禁止主文档贴实现代码**——必要时再单独写一份 `-代码实现.md`。

**5 个必含模块**（顺序固定）：
1. **架构 / 模块设计** — 默认 Mermaid 类图/组件图
2. **关键流程时序图** — 默认 Mermaid sequenceDiagram，优先时序图而非流程图
3. **关键逻辑** — 难点 + 候选方案 + 选择理由（含为什么不选另外的）
4. **接口说明** — 契约（path / method / schema / 错误码），不写实现
5. **遗留问题** — P1/P2/P3 编号 + 重开触发条件

## 何时触发

- ✅ "出个技术方案" / "方案设计" / "设计一下 XX 模块"
- ✅ 需求描述 + 参考截图 / 链接 / 类似产品
- ✅ "帮忙实现 XX 功能" 且复杂度需要先写方案
- ✅ 跨 ≥2 模块的重构，需要团队对齐
- ❌ 小修改（<50 行/单文件）→ 直接改
- ❌ Bug 修复 → 用 `superpowers:systematic-debugging`
- ❌ 已有 brainstorming spec → 直接走 `superpowers:writing-plans`

## 它交付什么

| | |
|---|---|
| 输出位置 | 项目根的 `方案设计/`（**不是** `docs/`） |
| 主文档 | `方案设计/YYYYMMDD-方案名称.md`（5 模块齐全） |
| 代码实现（可选） | `方案设计/YYYYMMDD-方案名称-代码实现.md`，仅当包含非平凡算法或难写对的骨架时 |
| 绘图工具 | **默认 Mermaid**（直接写 ```mermaid 代码块）；不够用再回退 `superpowers:uml` |

## 跟其它技能怎么衔接

- 上游可衔接 `superpowers:brainstorming`（想法未明时先 brainstorm）
- 下游接 `superpowers:writing-plans`（方案审核通过后落成可执行 plan）
- 方案产出归 [`dum-knowledge-base-build`](../dum-knowledge-base-build/) 搭的 `docs/tech-design/` 也可（取决于项目约定，但本技能默认 `方案设计/`）

## 完整工作流

5 模块详细约束、文档/代码分离的禁止与允许清单、红旗自检（出现即停）、常见错误，详见 [`SKILL.md`](SKILL.md)。
