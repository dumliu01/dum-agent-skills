# Dum Doc Reconcile

## Overview

按**时间范围**，用 `docs/modify_history/` 的记录（**兼查 git**）把 `docs/architecture/` 与 `docs/方案设计/` 两个目录的设计文档**校回与现状一致**。它是 [dum-session-summary] 的下游：前者每次会话产出"受影响的设计文档（待校对）"，本技能定期把这些待校对项真正落地修正。

**核心原则（四条，缺一不可）：**
1. **报告先行** —— 先产出"漂移报告"，**停下等用户确认**，再动文档。设计文档是精雕内容，未经确认的重写会抹掉有意的细节。
2. **modify_history 为主、git 兼查** —— 以各篇"受影响的设计文档"节为主线，再用同范围 git diff 补漏（找出没被任何记录提到、却影响了这两个目录所述内容的改动）。
3. **只改语义 prose，不碰自动区** —— `architecture/*.md` 里 `<!-- AUTO-GENERATED -->` 目录树由脚本维护，绝不手改。
4. **改完打校对水印** —— 让下次回看一眼看出每篇校到哪了。

## 触发判断

**该触发**：用户说"按时间范围/根据修改记录校对设计文档"、"让 architecture / 方案设计 保持最新有效"、"定期对账文档"、"reconcile docs"。

**不该触发**：要更新代码目录树（那是 `scripts/docs/update_architecture_*.py` 的活）；要写新设计方案（用 dum-solution-design）；要总结某次会话改了什么（用 dum-session-summary）。

## 工作流（四相，Phase 2 后必须停下）

### Phase 1 · 取数（限定时间范围 + 两个目标目录）
1. 解析时间范围为绝对日期（"最近两周"等相对说法先换算成 `YYYY-MM-DD`）。
2. 收集 `docs/modify_history/` 中**日期落在范围内**的记录；逐篇抽取「受影响的设计文档（待校对）」+「改动清单」+「关键决策」。
3. **git 兼查**：`git log --since=<start> --until=<end>`（或范围内 commit）取 diff，找出触及这两个目录所述模块/路径、却**没被任何 modify_history 提到**的改动。
4. 汇总成「待校对清单」（去重；**只保留 `docs/architecture/*` 与 `docs/方案设计/*` 两目录内**的文档；其它目录即使被提到也不在本技能范围）。

### Phase 2 · 漂移报告（报告先行）→ 停
对待校对清单里每个文档：读它，逐条比对"文档现述 vs 当前事实"，按下方模板列出漂移点，区分【事实性·可直接改】与【需判断·要你拍板】；无漂移的也列"已核对·无漂移"。
**输出报告后停下，等用户确认/挑选要改的项**。不要跳到 Phase 3。

### Phase 3 · 应用（确认后才做）
按确认结果改文档，遵守：
- **不碰** `architecture/*.md` 的 `<!-- AUTO-GENERATED:START/END -->` 区；若目录树明显过时，**提示**用户重跑 `python scripts/docs/update_architecture_*.py`，而不是手改。
- 保 `方案设计` 的 **5 模块结构**（架构/时序/关键逻辑/接口/遗留）与**文档代码分离**（不塞示例代码；接口签名、配置项名、commit hash 可以）。
- 改"遗留问题"时，已解决项标 `(已解决 YYYY-MM-DD)` 并补新引入的权衡，不要直接删历史。

### Phase 4 · 水印 + 收尾
给每个改过/核对过的文档打/更新校对水印（见下）；末尾报告：改了哪些文档哪几处、哪些仍需人工后续。

## 校对水印约定

在文档 H1 标题**正下方**放一行（已存在则原地替换，保持每篇唯一）：

```markdown
> 🗓️ 文档校对：已对齐至 2026-05-23（依据 modify_history + git 至 `9ac52f0`）— dum-doc-reconcile
```

- 日期用校对当天；`9ac52f0` 为本次覆盖到的最新 short hash。
- 下次校对从"上一条水印日期之后"接着扫，避免重复劳动。

## 漂移报告模板

```markdown
# 文档校对漂移报告 · 范围 2026-05-01 ~ 2026-05-23

来源：modify_history N 篇（列文件名）；git 兼查 commit 范围 <a>..<b>。

## docs/方案设计/20260419-Stage1bOnlineDataSources.md
| # | 位置(节) | 现述（过时） | 应为 | 依据 | 类别 |
|---|---|---|---|---|---|
| 1 | 关键逻辑 | quote_service 直连 ClickHouse 无缓存 | 缓存优先、miss 回源 | 2026-05-12 记录 / 9aa11bc | 事实性·可直接改 |
| 2 | 遗留问题 | "暂无缓存压力大" | 标记已解决 + 新增一致性窗口权衡 | 同上 | 需判断 |

## docs/architecture/backend.md
| # | 位置 | 现述 | 应为 | 依据 | 类别 |
|---|---|---|---|---|---|
| 1 | §3 核心模块 | core/data 无 cache 子模块 | 补 core/data/cache | 2026-05-12 记录 | 事实性·可直接改 |
| - | AUTO 区 | 树过时(05-10) | 不手改 → 提示重跑生成脚本 | — | 提示 |

## 已核对·无漂移
- docs/方案设计/20260420-Stage3IBBroker.md（范围内改动未触及）

> 请确认/挑选要应用的项；确认后我再改文档并打校对水印。
```

## 关键逻辑 / 难点

- **范围边界**：优先用户给的绝对范围；没给就问，或以"各文档上一条校对水印日期之后"为起点。
- **兼查的意义**：modify_history 可能漏写"受影响文档"；git diff 是兜底事实源——凡触及设计文档所述模块/路径的 commit，都要回看对应文档是否需改。
- **报告先行是硬性 gate**：Phase 2 必须先交报告。直接改 = 违反本技能（见红旗）。
- **AUTO 区与 prose 分界**：architecture 文档"目录树"归脚本，"模块语义/技术栈/流程"归人——本技能只动后者。

## 红旗 —— 出现即停，回到正轨

- 还没出漂移报告就开始改文档 → 停，先出报告等确认。
- 手改了 `architecture/*.md` 的 AUTO-GENERATED 区 → 撤销，改为提示重跑脚本。
- 往 `方案设计` 里塞示例代码、或打破 5 模块结构 → 撤销。
- 把范围外目录（如 guides/、product/）也改了 → 撤销，超出本技能范围。

## 常见错误

| 错误 | 后果 | 修正 |
|---|---|---|
| 跳过报告直接改 | 抹掉有意细节、不可控 | 报告先行，Phase 2 后停下确认 |
| 只看 modify_history 不查 git | 漏网改动留下漂移 | 同范围 git diff 兼查 |
| 手改 AUTO-GENERATED 目录树 | 下次脚本运行被覆盖、冲突 | 不碰，提示重跑生成脚本 |
| 不打/不更新水印 | 下次回看重复劳动、不知校到哪 | H1 下唯一一行水印，原地替换 |
| 删掉"遗留问题"历史 | 丢失设计沿革 | 标 `(已解决 日期)` + 补新权衡 |
| 校对范围外的目录 | 越权改动 | 只限 architecture 与 方案设计 |
